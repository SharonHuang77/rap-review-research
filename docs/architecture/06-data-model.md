# 6. Data Model & Persistence Architecture

## 6.1 Overview

This chapter defines the logical data model of the AI Code Review Experiment Platform and its mapping to persistent storage.

The platform is designed using a domain-driven approach. Domain entities are defined independently of the underlying database technology to ensure that business logic remains portable and maintainable.

Persistence is treated as an implementation detail rather than a core architectural concern.

---

# 6.2 Design Principles

The data model follows the following principles.

- Experiment-Centric
- Immutable Research Artifacts
- Idempotent Persistence
- Separation of Domain and Storage
- Replayability
- Versioned Experimental Data

Every stored entity represents either:

- an immutable research artifact, or
- an execution record generated during experimentation.

---

# 6.3 Domain Model

The platform consists of seven primary domain entities.

```text
                    Experiment
                         │
        ┌────────────────┼────────────────┐
        │                │                │
        ▼                ▼                ▼
   PR Snapshot      Review Result     Execution Log
        │                │
        │                ▼
        │            Finding
        │                │
        │                ▼
        └────────────► Metric
```

The Experiment is the aggregate root.

---

# 6.4 Entity Relationships

```text
Experiment
    │
    ├── references one PR Snapshot
    │
    ├── produces one Review Result
    │
    ├── produces many Findings
    │
    ├── produces one Metrics Record
    │
    └── produces many Execution Logs
```

All entities are linked through the Experiment ID.

---

# 6.5 Experiment

## Description

The Experiment is the primary entity of the platform.

Every experiment represents a single execution of one review architecture on one immutable PR Snapshot.

### Attributes

| Attribute | Description |
|------------|-------------|
| experimentId | Unique identifier |
| snapshotId | Associated PR Snapshot |
| architecture | Review architecture |
| modelVersion | LLM model |
| promptVersion | Prompt set |
| workflowVersion | Workflow implementation |
| evaluationVersion | Evaluation engine version |
| status | Current execution state |
| createdAt | Creation timestamp |
| completedAt | Completion timestamp |

---

### Lifecycle

```text
CREATED

↓

QUEUED

↓

RUNNING

↓

VALIDATING

↓

COMPLETED
```

Failure path

```text
RUNNING

↓

FAILED
```

---

# 6.6 PR Snapshot

## Description

A PR Snapshot is an immutable representation of a pull request imported from GitHub.

It guarantees that future experiments operate on identical input.

### Attributes

| Attribute | Description |
|------------|-------------|
| snapshotId | Unique identifier |
| repository | Source repository |
| prNumber | Pull Request number |
| commitHash | Commit SHA |
| title | Pull Request title |
| description | Pull Request description |
| rawDiff | Raw unified diff |
| changedFiles | Modified files |
| changedLines | Modified line ranges |
| importedAt | Import timestamp |

Snapshots are immutable after creation.

---

# 6.7 Review Result

The Review Result stores the final validated output returned by a review architecture.

### Attributes

| Attribute | Description |
|------------|-------------|
| resultId | Unique identifier |
| experimentId | Parent experiment |
| architecture | Review architecture |
| summary | Overall review summary |
| riskLevel | Estimated risk |
| findingCount | Number of findings |

The Review Result references individual Findings rather than duplicating them.

---

# 6.8 Finding

A Finding represents a single issue detected during code review.

### Attributes

| Attribute | Description |
|------------|-------------|
| findingId | Unique identifier |
| experimentId | Parent experiment |
| title | Finding title |
| category | Issue category |
| severity | Severity |
| file | File path |
| line | Line number |
| description | Explanation |
| recommendation | Suggested fix |
| confidence | LLM confidence |

Findings are immutable once validated.

---

# 6.9 Metrics

Metrics are generated by the Evaluation Engine after experiment completion.

### Synthetic Metrics

- Precision
- Recall
- F1 Score
- Localization Accuracy

### Real PR Metrics

- Evidence Score
- Architecture Agreement
- Finding Count

### Efficiency Metrics

- Latency
- Token Usage
- API Cost
- LLM Calls
- Agent Messages

---

# 6.10 Execution Log

Execution Logs record workflow execution.

### Stored Information

- Start time
- Finish time
- Agent execution order
- Retry attempts
- Errors
- Validation failures

Execution Logs support debugging but are excluded from experimental evaluation.

---

# 6.11 Aggregate Root

The Experiment acts as the aggregate root.

```text
Experiment

├── Snapshot

├── Review Result

├── Findings

├── Metrics

└── Execution Logs
```

All related entities are retrieved using the Experiment ID.

---

# 6.12 Persistence Strategy

The logical model maps to persistent storage as follows.

| Entity | Storage |
|----------|----------|
| Experiment | DynamoDB |
| PR Snapshot | DynamoDB |
| Findings | DynamoDB |
| Metrics | DynamoDB |
| Execution Logs | DynamoDB |
| Raw Diff | Amazon S3 |
| Raw Agent Outputs | Amazon S3 |

Large text artifacts are stored in S3 to reduce database size.

# 6.15 Versioning

Every experiment records version information.

```
Model Version

Prompt Version

Workflow Version

Evaluation Version
```

This guarantees reproducibility.

---

# 6.16 Immutability Rules

The following entities are immutable.

- PR Snapshot
- Review Result
- Finding
- Metrics

Only Experiment Status may change during execution.

---

# 6.17 Data Retention

Research artifacts should never be overwritten.

Instead

```
Experiment V1

↓

Experiment V2
```

Each experiment receives a unique identifier.

Historical experiments remain available for future analysis.

---

# 6.18 Repository Pattern

Business logic never accesses DynamoDB directly.

Instead

```text
ExperimentService

↓

ExperimentRepository

↓

DynamoDB
```

Every entity has its own repository interface.

This allows storage implementation to change without affecting business logic.

---

# 6.19 Summary

The data model is centred around the Experiment aggregate, ensuring that every experiment is reproducible, replayable, and versioned.

By separating the domain model from the persistence implementation, the platform remains portable across storage technologies while preserving a clean architectural design.
